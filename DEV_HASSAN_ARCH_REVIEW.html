Architectural Review: development Branch

Reviewer: Indrani Sen (Head of AI Engineering UK&I, Cognizant AI & Analytics Practice)

Date: June 9, 2026

Scope: Follow-up review of the development branch to assess progress on the June 1 architectural roadmap.

1. Where We Stand Today

Hassan, incredible work on the development branch over the past week. You've successfully taken the initial prototype and transformed it into a resilient, enterprise-grade foundation.

The migration to a monorepo structure (properly decoupling apps/ from packages/) provides exactly the structural discipline we need to scale this platform across global markets. More importantly, your implementation of the Zero-Trust On-Behalf-Of (OBO) architecture is a massive win. By enforcing Entra ID JWT validation via the requireAuth middleware, you have completely mitigated our biggest compliance and cost-exposure risks. The control plane is now secure, modular, and ready for production.

With this rock-solid scaffolding and security boundary firmly in place, we are perfectly positioned for the most exciting phase of the project: unlocking the true autonomy of the swarm. The next evolution is shifting our paradigm from a secure web application into a dynamic, intent-driven Agentic platform. The roadmap below outlines the architectural shifts required to get us there—specifically focusing on semantic data access, responsive streaming, and code-first App-Layer orchestration.

2. The Next Big Unlocks

2.1. Unchaining the Data Layer (Genie MCP Integration)

Priority: Critical Urgency: Immediate

Currently, dbsql is running off those hardcoded .sql files. It's solid for the initial demo, but it completely handcuffs the LLM. The agent can't answer anything outside of the queries you've pre-written.

Next Steps:

We need to deprecate the queries/ folder. However, we shouldn't let the agent write raw SQL on the fly—that's far too non-deterministic and opens us up to hallucinations or runaway compute costs. The right approach is to wire up the relevant Databricks Genie MCPs directly into the Foundry agent definition. Let the agent use the Genie tools to safely query our curated semantic models and trusted assets. This gives us the flexibility of an LLM with the deterministic accuracy the trading floor requires.

2.2. Making the UI Feel Alive (Streaming)

Priority: High Urgency: High

I noticed foundryProxy.ts is still awaiting the full payload before sending the HTTP response back. We can't do this in production. When the agent starts writing its own SQL via MCP, inference is going to take 10-30 seconds. Azure App Service will drop the connection, and from the user's perspective, the UI will just hang.

Next Steps:

Switch the Foundry SDK call to stream: true and move to Server-Sent Events (SSE). You need to flush the headers immediately (res.flushHeaders()) and stream the chunks back to the React app as they arrive. Let me know if you need a hand with the Azure Foundry stream configuration.

2.3. Reducing Cognitive Load for the Traders

Priority: High Urgency: Medium

The brief.tsx component is getting too heavy. A single vertical scroll forces the traders to hunt for the numbers they care about, and the full page swap to the Chat screen breaks their data context.

Next Steps:

Let's break this into a tabbed view (Fluent 2 components are fine for this). Keep the top KPI strip persistent, and put Sales, Margin, and Logistics into their own tabs. Also, pull the Chat into a persistent side panel so they can ask questions while looking directly at the data.

3. Aligning with Frontier Agentic Patterns (Azure-Native)

To ensure Aurora AI isn't just a basic LLM wrapper, we need to adopt the latest architecture patterns from Anthropic, OpenAI, and Microsoft. Crucially, because we are an Azure shop, we must execute these patterns using native Azure components to maintain our compliance and security boundaries.

3.1. Predictability via Code-First Workflows

Anthropic's latest research emphasizes that for deterministic enterprise tasks (like generating the Monday Brief), we shouldn't rely on a single, massive zero-shot prompt. Instead, we should use an Evaluator-Optimizer workflow (Draft -> Evaluate -> Refine).

Next Steps:

Crucially, do not offload this logic into a drag-and-drop cloud GUI like Azure Prompt Flow. Doing so fragments our business logic out of version control. We must maintain a "Lean Foundry Agent" pattern—Azure Foundry acts purely as the secure inference engine. The complex, versioned business logic (the prompts and the Evaluator-Optimizer orchestration loop) must be written natively as TypeScript inside our apps/api layer. The app layer owns the complexity; the cloud provides the compute.

3.2. Multi-Agent Handoffs

OpenAI's Swarm architecture proves that a single omniscient copilot doesn't scale. An agent trying to understand marketing ROI, WMS logistics, and gross margin simultaneously will inevitably hallucinate due to context overload.

Next Steps:

Utilize the Azure AI Agent Service to define specialized, narrow-scope agents. We need a lightweight "Router" agent that intercepts the user's chat. If it's a stock query, it hands off to the logistics-copilot. If it's profitability, it routes to the margin-analyst. This keeps system prompts small and token spend low.

3.3. Deep Observability

Enterprise agents require deep tracing to understand exactly why an agent made a decision or called a specific tool.

Next Steps:

Wire the orchestration layer directly into Azure Application Insights and Azure AI Foundry Tracing. Every MCP tool call, prompt execution, and agent handoff must be logged with its exact token usage and latency. This is the only way we can debug the "black box" when a trader reports an incorrect response.

4. Closing Thoughts

The refactoring this week was exactly what we needed to get the foundations right. Let's get the Databricks Genie MCP and Azure streaming wired up next so we can actually show off the agentic capabilities to the stakeholders. Keep up the good work.

Architectural Review: Aurora AI (dev-hassan Branch)

Reviewers:

  • Indrani Sen (Head of AI Engineering UK&I, Cognizant AI & Analytics Practice)
  • Aegis (Cognitive Code Sentinel, powered by Claude Opus 4.6)
  • Archytas (Topology & Systems Architecture Agent, powered by Claude Opus 4.6)

Date: June 1, 2026

Scope: Architecture, modularity, security posture, and user experience of the Aurora AI branch—assessed against frontier Agentic AI standards within the Azure AI Foundry ecosystem.

1. Where We Stand

Hassan has done a genuinely impressive job getting Aurora to this point. The domain logic works, the Databricks integration is solid, the brief renders real trade data, and the whole thing runs end-to-end on Azure App Service. For a V0, this is exactly what you want to see—working software that proves the concept and unblocks the business conversation.

What follows is not a list of things that are wrong. It is a roadmap for turning a strong proof-of-concept into something we can confidently put in front of 80+ traders across five markets. The patterns below are drawn from how frontier AI organisations (Anthropic, Google, OpenAI) structure their production systems, and from Microsoft's own guidance on building enterprise agentic applications.

2. Observations and Opportunities

2.1. Moving from Static SQL to Agentic Data Access

Priority: High Urgency: High

Hassan built the data layer around two modules: server/dbsql/client.ts (a thin wrapper over the Databricks SQL REST API) and server/dbsql/pull.ts (the orchestrator that maps each business prompt to a bundle of SQL files). There are 10 handwritten .sql queries on disk—sales_daily.sql, brands_top10.sql, margin_daily.sql, and so on—each executed with fixed parameter bindings (:market, :week_start, :week_end). The pull.ts file uses a switch statement to route prompts like PROMPT_BRIEF_SALES_DATA to the right SQL bundle.

The queries themselves are well-written. The multi-CTE joins across fact_billed_sale_and_return_v1, dim_product_v1, and dim_country_v1 are clean and performant. But they are static—and that is the constraint. If a trader asks "what is the return rate by price band for new customers?", the system has nowhere to go because no .sql file covers that combination. The model's intelligence is capped at whatever questions Hassan anticipated when writing the queries.

The natural next step is to connect the Databricks catalogue to the LLM via the Model Context Protocol (MCP)—specifically through Databricks Genie as an MCP tool inside the Azure AI Foundry agent definition. This lets the agent inspect the schema, write its own SQL, run it, and self-correct if it hits a syntax error. Hassan would no longer need to maintain a growing library of .sql files, and the system could handle genuinely novel questions from the business.

2.2. Separating Infrastructure from Business Logic

Priority: High Urgency: Medium

Right now, server/proxy.ts is doing a lot of heavy lifting. In 377 lines, it handles: Azure Managed Identity token acquisition (lines 42–66), HTTP proxying with socket-error retries to the Foundry endpoint (lines 94–165), a JSON cache API with a hardcoded MARKET_TO_REGION_CODE dictionary (lines 171–177), brief serving from the local filesystem (lines 210–264), a week-listing endpoint (lines 269–320), a facts endpoint (lines 322–345), and static SPA hosting (lines 347–361). That is six distinct responsibilities in one file.

This works fine at the current scale, and keeping everything together made it easier for Hassan to ship quickly. But as we add new markets, new agent capabilities, or new data sources, modifying this file becomes increasingly risky—a change to how briefs are cached could accidentally break the auth flow, and vice versa.

The goal is to pull these apart: a thin Control Plane that handles authentication, token management, and generic routing, and a separate Capability Plane where domain logic (brief caching, agent orchestration, market mapping) lives as pluggable modules. That way, Hassan—or any developer who follows—can add a new agent capability without touching the auth middleware. The Azure AI Foundry Responses API is designed exactly for this pattern: the proxy becomes a simple pass-through, and the intelligence lives in Foundry.

2.3. Streaming Instead of Blocking

Priority: Medium Urgency: Medium

The Foundry proxy currently reads the entire LLM response into memory with await response.text() (line 141) and sends it back to the browser in one shot via res.send(responseBody) (line 152). Hassan has already set a 300-second timeout (TIMEOUT_MS = 300_000, line 30), which tells me he knows these agentic calls can take a while.

The problem is that during those 300 seconds, the browser gets nothing—no progress, no partial content, no sign that the system is working. From the user's perspective, the screen just hangs. Worse, Azure App Service load balancers typically drop idle connections after about 4 minutes, so longer agent runs risk a 504 before the response even arrives.

The fix is straightforward: switch to Server-Sent Events (SSE). Instead of buffering the whole response, the proxy streams delta chunks as the model generates them. The user sees text appearing in real-time, the connection stays alive, and the UX goes from "is it broken?" to "I can see it thinking." Every major AI product—ChatGPT, Gemini, Claude—uses this pattern for exactly this reason.

Implementation Guidance for Hassan:

To wire this up cleanly with the upcoming MCP integration, you'll need three pieces:

  1. The Backend Config: In proxy.ts, instead of res.send(), set the headers to Content-Type: text/event-stream, Cache-Control: no-cache, and Connection: keep-alive. You must call res.flushHeaders() immediately to force the load balancer to open the stream and prevent a 504 timeout.
  2. The Foundry API Call: Pass stream: true to the Azure AI Foundry SDK. As chunks arrive from Foundry, write them directly to the Express response stream using res.write(`data: ${JSON.stringify(chunk)}\n\n`).
  3. Tool/MCP Telemetry: It's not just text! As the agent invokes MCP tools (like the Databricks Genie), stream those events down too. Emit tool_call chunks when the agent starts a query, and tool_result chunks when it finishes. This lets the React frontend render a beautiful "Agent is querying Databricks..." loading state while the LLM thinks.

2.4. Locking Down the Front Door

Priority: Critical Urgency: Immediate

Hassan has done a solid job on outbound auth. The DefaultAzureCredential flow for acquiring Managed Identity tokens is well-implemented, and the API-key fallback for local development is pragmatic. Token caching with auto-refresh (lines 60–63) is a nice touch.

The gap is on the inbound side. The /api/foundry route (line 94) accepts requests from anyone—no JWT validation, no user identity check, no authentication middleware whatsoever. The frontend even has a MockDataBadge labelled "Demo user" in App.tsx (lines 127–129), which confirms user identity is hardcoded. If this endpoint is reachable from the internet, anyone can call it and run up the Azure AI token bill—or worse, access sensitive trade data.

Before this goes to production, we need a proper Zero-Trust On-Behalf-Of (OBO) flow. The client acquires an Entra ID JWT via the Teams SDK, sends it as a Bearer token, and the proxy validates the signature against Microsoft's JWKS endpoint before forwarding anything to Foundry. This is table stakes for any enterprise app that calls a paid AI service.

2.5. Rethinking the User Experience

Priority: High Urgency: Medium

The Monday Brief currently renders as a single, vertically scrolling page inside brief.tsx (717 lines). Everything—the headline block, region selector, daily roll-up table, four metric grids (Demand, Mix, Sales by price type, Margin), the editorial "Why" commentary, suggested actions, a follow-up input, and a "Browse by week" archive—is stacked top to bottom in one long column. The navigation rail in App.tsx offers just two routes: "Monday Brief" and "Chat", and switching between them is a full page swap.

For an initial build, this editorial "newspaper" layout makes perfect sense—it is easy to reason about and easy to build. But for an interactive trading tool that 80+ users will rely on daily, it creates some real friction:

  • Scrolling fatigue. Traders looking for margin data have to scroll past the headline, the daily table, the demand grid, the mix grid, and the sales grid before they reach what they need. The Nielsen Norman Group's research on dashboard design shows that users abandon dense, scroll-heavy layouts when key metrics are not visible within the initial viewport.
  • No responsive breakpoints. The daily roll-up table in daily.tsx sets minWidth: 900 and falls back to horizontal scrolling on smaller screens. There are no CSS media queries anywhere in styles.css. Traders checking the brief on a tablet during a morning commute will have a frustrating experience.
  • Chat as a separate world. The current Brief-to-Chat transition is a full page swap (route === 'chat' in App.tsx, line 232). If a trader spots an anomaly in the margin data and wants to ask a follow-up question, they lose sight of the data entirely when they switch to chat. A persistent, collapsible chat panel alongside the brief would be far more natural.
  • Data density without escape hatches. The daily table is impressively comprehensive—TY, Budget, MYF, LY, LW, L4W, plus delta rows—but showing everything at once can overwhelm rather than inform. A sensible default would be TY plus one comparison row (vs LY), with a "Show all comparisons" toggle for power users who want the full matrix.
  • Metric cards could be more interactive. The 4-column metric grid uses a fixed gridTemplateColumns: 'repeat(4, 1fr)' layout. Each cell shows a value, a sparkline, and two delta comparisons. This is good, but the cells are passive—hovering or clicking does nothing. Elevating each metric to an interactive card that reveals a mini-breakdown on hover (or tap) would let traders drill into the numbers without leaving the page.

The north star here is a tabbed dashboard where the headline KPI strip stays pinned at the top, and the remaining content is organised into tabs ("Daily Breakdown" | "Demand & Mix" | "Margin" | "Commentary & Actions"). Hassan has already built all the underlying components—this is primarily a layout refactor, not a rebuild.
